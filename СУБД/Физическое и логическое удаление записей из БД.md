# Физическое и логическое удаление записей из БД

## 2 вида удаления записей из таблиц БД

Существует 2 вида — физическое и логическое.

Критерий|Физическое удаление|Логическое удаление
--------|-------------------|-------------------
Назначение|<ul><li>Удаление данных спамеров, т.к. они не имеют ценности<li> Удаление данных, созданных явно по ошибке<li>Удаление очень старых данных, уже потерявших ценность|<ul><li>Удаление неактуальных данных, имеющих историческую ценность
Основной принцип работы|Записи удаляются безвозвратно навсегда|Ставится только отметка (флаг), что запись "удалена", т.е. это аналог корзины
Преимущества|Освобождается место на диске (несущественное преимущество)|<ol><li>Информация не теряется. Удаление можно откатить и вернуть как было (люди ошибаются)<li>В случае претензий клиента по оказанным услугам можно "поднять" историю и отстоять позицию компании<li>Возможен аудит/статистика/аналитика "удалённых" записей с целью повышения качества предоставляемых услуг
Недостатки|Автоматически удаляются связанные с сущностью производные данные или связь с ними разрывается. В последнем случае понять взаимосвязь между данными практически невозможно.<br>Примеры:<ol><li>При удалении соискателя/компании не должны автоматически удаляться все его резюме/вакансии и отклики на них, т.к. у компании/соискателя должна сохраниться история взаимодействия откликов на резюме /вакансию, история звонков, оплаты услуг и др.<li>При удалении вакансии/резюме не должны удаляться отклики, т.к. у соискателя/компании должна сохраниться история взаимодействия откликов на вакансии/резюме, история звонков, оплаты услуг и др.<li>При удалении сотрудников (менеджеров, модераторов, администраторов и др.) должна сохраняться их активность, чтобы понимать кто, когда и что делал|<ol><li>Логика кода усложняется (детали см. ниже)<li>"Удалённые" записи занимают место на диске (несущественный недостаток)

## 2 подхода к логическому удалению записей из таблицы БД

Критерий| Удалённые записи хранятся в той же таблице| Удалённые записи хранятся в другой таблице
--------|-------------------------------------------|-------------------------------------------
Основной принцип работы|В таблице `{table}` в поле `deleted_at` вместо `null` записывается дата-время удаления|Записи удаляются из таблицы `{table}` и добавляются в таблицу `{table}_deleted`
Преимущества|<ol><li>SQL запрос на "удаление" и откат (возврат к исходному состоянию) — простой, быстрый, надёжный: `UPDATE {table} SET deleted_at = NOW() WHERE id = {id}`<li>Целостность данных не нарушается.|<ol><li>Внедрить этот подход на большое кол-во готового кода проще, SQL запросы НЕ нужно доделывать.<li>В случае большого кол-ва "удалённых" записей (десятки процентов от общего кол-ва записей) общая скорость работы с таблицей `{table}` НЕ замедляется.
Недостатки|<ol><li>Внедрить этот подход на большое кол-во готового кода сложно, SQL запросы нужно доделывать: почти все SQL запросы должны явно иметь `условие WHERE deleted_at IS [NOT] NULL`<li>Все уникальные индексы должны явно иметь условие `WHERE deleted_at IS NULL` (несущественный недостаток)<li>В случае большого кол-ва "удалённых" записей (десятки процентов от общего кол-ва записей) общая скорость работы с таблицей {table} замедляется, если не используются индексы.|<ol><li>Процедура "удаления" и отката сложнее, т.к. запись нужно перемещать из одной таблицы в другую. Например, для "удаления" записи из {table} нужно делать триггер, а для отката — хранимую процедуру.<li>При изменении схемы таблицы `{table}` необходимо согласованно менять схему таблицы `{table}_deleted`<li>При наличии ссылающихся на таблицу внешних ключей переместить записи в другую таблицу проблематично. В зависимости от настроек FK связи разорвутся или связанные данные удалятся автоматически (что ещё хуже).
