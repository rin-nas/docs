# Где должна храниться бизнес-логика?

Статья **2023** года.

## Терминология

Из [Википедии](https://ru.wikipedia.org/wiki/%D0%91%D0%B8%D0%B7%D0%BD%D0%B5%D1%81-%D0%BB%D0%BE%D0%B3%D0%B8%D0%BA%D0%B0):

> **Бизнес-логика** — в разработке информационных систем — совокупность правил, принципов, зависимостей поведения объектов предметной области (области человеческой деятельности, которую система поддерживает). Иначе можно сказать, что бизнес-логика — это реализация правил и ограничений автоматизируемых операций. Является синонимом термина «логика предметной области» (англ. domain logic). > Бизнес-логика задает правила, которым подчиняются данные предметной области.

> Проще говоря, **бизнес-логика** — это реализация предметной области в информационной системе. К ней относятся, например, формулы расчёта ежемесячных выплат по ссудам (в финансовой индустрии), автоматизированная отправка сообщений электронной почты руководителю проекта по окончании выполнения частей задания всеми подчиненными (в системах управления проектами), отказ от отеля при отмене рейса авиакомпанией (в туристическом бизнесе) и т. д.

## Введение

В архитектуре современных веб-приложений три уровня:

1. Клиентское приложение (frontend) — уровень отображения данных и взаимодействия с пользователем посредством GUI. Это ПО, которое работает на устройстве (компьютере, смартфоне) пользователя.
   Например, графический редактор, игра, веб-браузер, мобильное приложение.
1. Серверное приложение (backend) — это уровень бизнес-логики.
1. Сервер базы данных (database) — это уровень хранения данных.

Исходный код бизнес-логики может храниться на одном уровне, а выполняться на другом.

№|Хранится|Выполняется|Пояснение
-|--------|-----------|---------
1|$\color{#00f}{\text{frontend}}$|$\color{#00f}{\text{frontend}}$|Бизнес-логика хранится и выполняется в клиентском приложении.
2|$\color{#0a0}{\text{backend}}$|$\color{#00f}{\text{frontend}}$|Клиентское приложение может запрашивать у сервера приложений ограничения для валидации введённых пользователем данных (минимальное и максимальное значение, регулярное выражение, сообщения об ошибках). Например, через OpenAPI JSON схему.
3|$\color{#0a0}{\text{backend}}$|$\color{#0a0}{\text{backend}}$|Бизнес-логика хранится и выполняется на сервере приложения.
4|$\color{#0a0}{\text{backend}}$|$\color{#f00}{\text{database}}$|Cервер приложений отсылает на сервер БД SQL запросы (по одному), которые там выполняются. На сервере приложения SQL запросы могут выстраиваться в логические цепочки: данные из результата одного запроса (например, числовые идентификаторы) могут участвовать в формировании следующего запроса.
5|$\color{#f00}{\text{database}}$|$\color{#f00}{\text{database}}$|Cервер приложений обращается на сервер БД к таблице, представлению (view), хранимой процедуре, функции. Бизнес-логика скрыта внутри этих объектов БД. При изменении данных в таблицах могут срабатывать триггерные функции. В хранимой процедуре или функции за 1 вызов могут выполняться несколько SQL запросов.

Большинство бизнес-логики должно храниться и выполняться на сервере приложения. Небольшую часть бизнес-логики выносят в клиентское приложение и БД.

Проценты очень приблизительные. Их сумма > 100, т.к. есть дублирование логики на разных уровнях.

## Почему нужно хранить бизнес-логику в приложении на сервере?

### Для масштабирования производительности

Приложение гораздо проще и дешевле [масштабируется горизонтально](https://ru.wikipedia.org/wiki/%D0%9C%D0%B0%D1%81%D1%88%D1%82%D0%B0%D0%B1%D0%B8%D1%80%D1%83%D0%B5%D0%BC%D0%BE%D1%81%D1%82%D1%8C), чем БД. Достаточно запустить несколько копий приложения на разных серверах. Бесплатный PostgreSQL умеет масштабироваться по схеме master-slave. На запись возможен только один сервер, на чтение — много. Если нет запаса мощности, то БД станет узким местом на запись. Решения по схеме multi-master тоже существуют ([пример](https://postgrespro.ru/docs/enterprise/current/multimaster)), но они платные и зависимые от одной компании-разработчика. Ещё в БД применяют логическое масштабирование, когда на разных серверах размещают логически связанные наборы таблиц. Или используют распределённые таблицы (пример), когда на разных серверах размещают разные сегменты одной таблицы. При необходимости, организуют прямой доступ между серверами БД через [postgres_fdw](https://postgrespro.ru/docs/postgresql/current/postgres-fdw).

### Для масштабирования разработки и тестирования

Разрабатывать код приложения с системой управления версиями (СУВ) (например, GIT) удобнее, чем без неё. СУВ позволяет автоматически контролировать зависимости, разрабатывать параллельно в ветках (когда никто никому не мешает), смотреть на изменения кода во времени (в прошлом), делать ответвления и слияния веток. Для изменения БД во времени имеются SQL команды, которые позволяют накатить и откатить изменения в схеме и/или данных. Но если в СУВ для отката специально делать ничего не нужно, то для БД нужно писать код. Параллельная разработка в БД достигается с использованием ["тонких" клонов БД](https://postgres.ai/products/dblab_engine).

Упорядочивать, переиспользовать и поддерживать код в приложении проще, применяя разные [парадигмы программирования](https://ru.wikipedia.org/wiki/%D0%9F%D0%B0%D1%80%D0%B0%D0%B4%D0%B8%D0%B3%D0%BC%D0%B0_%D0%BF%D1%80%D0%BE%D0%B3%D1%80%D0%B0%D0%BC%D0%BC%D0%B8%D1%80%D0%BE%D0%B2%D0%B0%D0%BD%D0%B8%D1%8F), используя готовые библиотеки кода и frameworks. Языки [хранимых процедур](https://ru.wikipedia.org/wiki/%D0%A5%D1%80%D0%B0%D0%BD%D0%B8%D0%BC%D0%B0%D1%8F_%D0%BF%D1%80%D0%BE%D1%86%D0%B5%D0%B4%D1%83%D1%80%D0%B0) в БД были разработаны для <ins>базовых преобразований данных и их сохранения в непротиворечивом (согласованном) виде</ins>. Чтобы поддержать то, на что не хватало в SQL. Не нужно в БД делать то, что возможно сделать в серверном приложении. Например, форматировать телефонные номера, делать HTTP запросы к API микросервисов.


## Когда целесообразно выносить бизнес-логику в клиентское приложение?

### Для улучшения удобства использования (usability)
Например, на форме ввода имеются зависимости между полями ввода. Если пользователь отметил флаг (checkbox), то появляется дополнительный блок с несколькими полями ввода.

### Для ускорения взаимодействия с пользователем и экономии ресурсов сервера
Кол-во пользователей примерно равно количеству их персональных компьютеров, а количество серверов приложений ограничено. Некоторые нересурсоёмкие задачи можно возложить на компьютеры пользователей.

Пример 1. Проверку правильности введённых данных в формах ввода можно делать интерактивно по мере ввода или когда поле ввода потеряло фокус. При этом запросов к серверу будет не много, а только один после подтверждения заполнения формы ввода пользователем.

Пример 2. Форматирование телефонных номеров, корректировка и фильтрация текста. Поле ввода с автодополнением. Получение данных из внешних сервисов по API.

Пример 3. Финансовый калькулятор с расчётом дохода по вкладу или суммы ежемесячного платежа по кредиту. По бизнес- требованиям результат должен пересчитывается по мере ввода пользователем параметров в форму ввода (в реальном времени).

Данным, полученным от клиентского приложения, доверять нельзя (их легко подделать). Поэтому бизнес-логика дублируется на уровне серверного приложения.

Перенося часть бизнес-логики на клиентское приложение, снижаются затраты на серверные ресурсы.


## Когда целесообразно выносить бизнес-логику в БД?

Реляционные БД спроектированы <ins>для хранения</ins>, получения и модификации данных ([CRUD](https://ru.wikipedia.org/wiki/CRUD)) с максимально высокой эффективностью даже при параллельных конкурентных запросах. Важно, что БД соответствуют набору требований [ACID](https://ru.wikipedia.org/wiki/ACID).

Причины ниже являются [нефункциональными требованиями](https://habr.com/ru/post/231961/), кроме последних двух пунктов.

### Для исключительной надёжности
Много разных бизнес-процессов от разных владельцев разных серверных приложений
Так могло сложиться исторически: все (естественно, временно) годами "ходят" в одну БД: монолиты, микросервисы, старые и новые API, старые и новые "веб-админки", CRM, периодически запускаемые задачи по обслуживанию и очистке БД, задачи с изменениями БД.

При таком "зоопарке" в разных приложениях логика может отличаться. Каждый раз смотреть и анализировать код в нескольких приложениях долго, на практике так никто не делает. Так же затруднительно вносить правки в "замороженный" устаревший код, который уже никто не поддерживает. Отсюда риск, что бизнес логика где-то может работать некорректно и не завершаться ошибкой. В таком случае можно подстраховаться и добавить в БД ограничение, которое сделает невозможным некорректное состояние данных в БД. Такое решение можно рассматривать как временное (техдолг).

Много бизнес-процессов работают параллельно с конкурентным доступом к ресурсам
Тема с [ACID](https://ru.wikipedia.org/wiki/ACID) непростая, для минимизации ошибок лучше использовать штатные и надёжные механизмы в БД, чем писать свои в приложении. 
Примеры:
[раз](https://stackoverflow.com/questions/69357351/how-to-safety-add-new-money-operation-to-database),
[два](https://stackoverflow.com/questions/66667347/how-to-conditional-update-sql-tables),
[статья](https://www.enterprisedb.com/blog/postgresql-anti-patterns-read-modify-write-cycles).

Высокая стоимость устранения последствий сломанного бизнес-процесса
Что будет, если 100 млн. рублей перечислятся на не тот счёт и эти деньги уже обналичили? 
А если билет на самолёт, поезд, корабль на один и тот же рейс и посадочное место продать нескольким клиентам? Кто будет в этом разбираться и возмещать убытки?

Программиста и тестировщика, которые допустили ошибку, уволят за профнепригодность. А ещё могут привлечь к финансовой ответственности с пожизненными выплатами для компенсации убытков. Понимая цену такой ошибки, лучше перестраховаться.

Важны репутационные риски. Малоприятно, когда об инциденте узнает весь мир через СМИ или соцсети. Например, для банка репутация — самый ценный актив.

### Для экономии времени на разработку и поддержку кода
Если в БД код с использованием [ограничений](https://postgrespro.ru/docs/postgresql/current/ddl-constraints) или [триггеров](https://postgrespro.ru/docs/postgresql/current/sql-createtrigger) будет простым. А в серверном приложении — более сложным или нетривиальным.

Пример 1. Ограничение внешнего ключа (Foreign Key), которое проверяет ссылочную целостность двух связанных таблиц (когда значение в колонке одной таблицы существует в колонке другой таблицы).

Пример 2. [Триггер](https://github.com/rin-nas/postgresql-patterns-library#%D0%BA%D0%B0%D0%BA-%D1%81%D0%B4%D0%B5%D0%BB%D0%B0%D1%82%D1%8C-%D0%B0%D0%B2%D1%82%D0%BE%D0%BC%D0%B0%D1%82%D0%B8%D1%87%D0%B5%D1%81%D0%BA%D0%B8-%D0%BE%D0%B1%D0%BD%D0%BE%D0%B2%D0%BB%D1%8F%D0%B5%D0%BC%D0%BE%D0%B5-%D0%BF%D0%BE%D0%BB%D0%B5-updated_at), который при добавлении или обновлении строки таблицы сохраняет в колонке updated_at текущую дату и время, но при условии, что данные в любой колонке изменились (старое и новое значение отличается).

Пример 3. [Триггер](https://github.com/rin-nas/postgresql-patterns-library#%D0%BA%D0%B0%D0%BA-%D1%81%D0%B4%D0%B5%D0%BB%D0%B0%D1%82%D1%8C-%D0%B6%D1%83%D1%80%D0%BD%D0%B0%D0%BB%D0%B8%D1%80%D0%BE%D0%B2%D0%B0%D0%BD%D0%B8%D0%B5-%D0%B8%D0%B7%D0%BC%D0%B5%D0%BD%D0%B5%D0%BD%D0%B8%D0%B9-%D1%82%D0%B0%D0%B1%D0%BB%D0%B8%D1%86%D1%8B), который автоматически сохраняет историю изменения записей в таблице-журнале.

### Для экономии вычислительных ресурсов БД
Если в БД код работает намного быстрее, чем в приложении, а производительность важна.

Пример 1. Имеется огромный SQL запрос, который каждый раз передаётся по сети в БД и там разбирается. Это занимает драгоценное время и "засоряет" журнал БД в гигабайты. В таком случае лучше создать представление (view) или функцию в БД. Тогда нагрузка на сеть и БД будет меньше. При изменении схемы БД или медлительности запроса его можно оптимизировать без правок в коде приложения.

Пример 2. На один запрос клиентского приложения для формирования ответа серверное приложение делает десятки и сотни последовательных запросов к БД. Это будет работать долго в случае медлительности сети. Ускорить поможет CTE запрос, view, хранимая процедура или функция в БД, которая сразу подготовит все данные (например, в формате JSON) за 1 запрос.

### Для обеспечения безопасности доступа к данным
Если необходимо ограничить доступ к части данных в БД: схемам, таблицам, колонкам, строкам. Для этого создают view или функцию, которая принадлежит и будет выполняться от владельца, отличающегося от пользователя, под которым подключается приложение (чтобы код нельзя было изменить).

Пример. По соображениям безопасности у приложения нет доступа в БД к таблице с денежными счетами клиентов. Доступ к БД выполняется через функции. Первая возвращает остаток на счёте клиента. Вторая возвращает N последних транзакций по счёту клиента. Третья переводит деньги со счёта одного клиента на счёт другого клиента. Вызовы функций протоколируются в отдельную таблицу БД, доступ к которой ограничен. В журнал записываются: дата-время события, id сотрудника, id клиента, номер счёта клиента и др.

### Для реализации сервисных функций
Если необходимо обрабатывать данные непосредственно в БД <ins>в больших объёмах</ins>, периодически или разово:
1. Удалять ненужные, устаревшие, избыточные данные в таблицах БД для освобождения занятого места (см. [loop_execute()](https://github.com/rin-nas/postgresql-patterns-library/tree/master/procedures/loop_execute))
1. Пересчитывать значения (коэффициенты) или делать агрегацию (например, через materialized view или в обычных таблицах)
1. Валидировать схему БД при тестировании наката и отката изменений БД (см. "[Валидатор качества схемы БД PostgreSQL](https://github.com/rin-nas/postgresql-schema-quality-validator)")
1. Обезличивать персональные данные в тестовых БД (см. "Сервис детектирования и обезличивания ПД в PostgreSQL")

### Для реализации бизнес-требований
Если необходимо хранить SQL запросы <ins>на чтение данных</ins> в колонке таблицы БД.

Пример. В микросервисе рассылок или выгрузке продуктовых данных в XML или JSON файлы (feeds) требуется выборка данных из БД. Возможностей существующего конструктора запросов в веб-админке может оказаться недостаточно из-за сложности критериев выборки данных. Данные разнесены по связанным таблицам. Выборка строится не только по наличию, но и по отсутствию связанных данных; с учётом трансформации, агрегации и дедупликации данных. Важно, чтобы запрос к БД работал быстро. В таком случае можно написать SQL запрос и сохранить его через веб-админку в БД. Для большей безопасности такие SQL запросы выполняют под readonly пользователем БД, а ещё лучше делать запрос к реплике БД (она только на чтение).

Поместить такие SELECT запросы в исходный код проекта в *.sql файлы под GIT невозможно. Согласно бизнес-требованию нужно создавать и модифицировать выборки данных через веб-админку без выпуска и развёртывания (deploy) проекта.


## Ограничения в БД — это бизнес-логика

Бизнес-логика в БД может находиться в ограничениях, представлениях (view), хранимых процедурах и функциях (включая триггерные функции).

Задача БД не только надёжно хранить данные, но и <ins>не допустить противоречивость</ins> при сохранении данных (обеспечить согласованность) и возвращать ошибку в клиентское приложение при возникновении таких ситуаций. Для этого в БД есть механизм [ограничений](https://postgrespro.ru/docs/postgresql/current/ddl-constraints).

Виды ограничений:

1. Проверка сохраняемых данных на наличие значения (not null), диапазон допустимых значений, соответствие маске ввода, условия между значениями разных колонок. Примеры:
   1. числовые значения, которые не могут быть отрицательными:
      * цена товара д.б. >= 0 (ноль для подарков клиентам)
      * вес физического товара д.б. > 0 (для электронных товаров понятия веса вообще нет, для таких случаев есть NULL)
   1. устоявшиеся числовые границы:
      * широта: от −90 до 90; долгота: от −180 до 180
      * проценты для обозначения прогресса действия: от 0 до 100 (но проценты в финансах загонять в рамки не стоит, они могут быть отрицательные и больше 100)
   1. устоявшиеся форматы значений:
      * номер телефона должен соответствовать формату E.164 (иначе будут проблемы)
      * email, ИНН, КПП, ОГРН, БИК, СНИЛС, расчётный счёт, корреспондентский счёт и пр. должны соответствовать формату
   1. прочие невозможные ситуации:
      * дата-время заказа и дата рождения не может быть в будущем
1. Проверка на уникальность. Примеры:
   1. все справочники не могут иметь дубликаты в названиях
   1. список товаров на складе или список характеристик товара (вес, цвет, размер, ...) не должен иметь дубликаты
   1. cтанции метро в одном городе на одной линии не должны иметь одинаковых названий
1. Проверка ссылочной целостности (проверка корректности связей через внешние ключи). Примеры:
   1. резюме д.б. связано с соискателем
   1. платёжные реквизиты д.б. связаны с компанией
1. Проверка на исключения. Примеры:
   1. нельзя забронировать номер в гостинице или арендовать автомобиль так, чтобы период времени пользования пересекался с другим клиентом (пример решения в БД: раз, два)


Простые ограничения выше — это <ins>бизнес-логика</ins>, пусть даже примитивная. Существуют более сложные ограничения, которые возможно сделать с помощью триггеров. Примеры:
1. Неоплаченных заказов не должно быть больше суммы в N рублей.
1. Товар можно купить, если он есть на складе.
1. Товар можно купить, если у клиента хватает денег на счёте.
1. Не больше N штук товара/услуги одному клиенту (если товар/услуга акционная или дефицитная)
1. Нельзя отвязать от компании единственного пользователя-представителя.

### Как тестировать бизнес-логику в БД?
Бизнес-логика в БД покрывается уже существующими [функциональными тестами](https://habr.com/ru/post/549054/) для приложений. Эти тесты запускаются в CI (например, в GitLab).

<ins>Модульные тесты в БД</ins> нужны для функций, которые используются в ограничениях БД, [пример](https://github.com/rin-nas/postgresql-patterns-library/blob/master/functions/is/is_client_account.sql). 
Такие тесты выполняются в накате изменений БД после создания или изменения кода функций. Если разработчики не забывают добавлять тесты в изменение БД, то запускать тесты в CI не нужно.

### Как узнать в тесте, откуда ошибка: из приложения или из БД?
В продуманных приложениях в объекте ошибки имеется много полей. Cреди них поле message с общим текстом ошибки, которое возвращается всем желающим (в API, например). Детали ошибки не раскрываются по соображениям безопасности и избыточности.

Есть поле detail с детальным сообщением об ошибке. Оно интересно и полезно только разработчикам. Здесь можно показывать конкретные значения (числа/строки/прочее). Поле возвращается всем желающим в тестовой среде, но не доступно в производственной среде (запись только в журнал ошибок). В этом поле можно передать объект ошибки от БД. У этой ошибки будет свой message, detail, code и другие поля. Эти поля будут однозначно идентифицировать место ошибки в БД. PostgreSQL [позволяет](https://postgrespro.ru/docs/postgresql/current/plpgsql-errors-and-messages#PLPGSQL-STATEMENTS-RAISE) возвращать ошибку с произвольным кодом (например, из [триггера-ограничения](https://postgrespro.ru/docs/postgresql/current/sql-createtrigger)) и полем hint, в котором подсказывают (!), как решить проблему.

В итоге, при получении ошибки в функциональном тесте приложения будет известно, откуда она, со всеми деталями.

Если в цепочке взаимодействующих микросервисов или компонентов одного приложения (далее звено) возникает ошибка, то она последовательно передаётся до последнего звена. Не хочется делать обратную последовательную трассировку, чтобы узнать, на каком звене сломалось и почему. Чтобы сразу получить информацию, от звена к звену передают ошибку с полем detail, которое может быть объектом ошибки предыдущего звена или строкой для первого звена.
